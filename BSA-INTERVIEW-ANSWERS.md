# Business Systems Analyst — Interview Answers (Bell Canada Subscription Platform)

> Grounded in ~2 years as a BSA on Bell Canada's Subscription Management Platform
> (60+ Go microservices + a Next.js micro-frontend, integrating Netflix, Disney+,
> Bell Media, Bango, Radio-Canada). Source: `src/data/bsa-cheatsheet.ts`.

## 1. Describe a situation where you had to analyze a complex business process.
The most complex process I analyzed was the end-to-end subscription lifecycle — add, activate, grace period, cancel, reverse — across 5 streaming providers, each with different provisioning protocols and grace windows (Netflix 3 days, Bango 7). The existing docs described a simple "customer adds a subscription" flow, but the reality was 8 microservices, 3 databases, Kafka event chains, and saga compensation logic. I traced the code through reseller-service and built a 6-step orchestration spec — session validation → qualification → order write → merchant provisioning → Kafka publish → audit — documenting the service, payload, failure mode, and compensation for each step, plus a state machine of every status transition. That surfaced edge cases the product team hadn't considered, like reversing a cancellation mid-grace-period. The spec became the team's single source of truth: QA derived 17 test scenarios from it, and when we onboarded Radio-Canada it let us pinpoint that only the adapter layer needed customization — onboarding took 2 weeks instead of the projected 5.

## 2. How do you prioritize and manage requirements in a project?
I prioritize on three dimensions. First, business value and urgency — merchant launch dates were contractual (Disney+ had a firm go-live), so those automatically outranked internal improvements. Second, technical dependencies — I map them in orchestration specs and sequence stories so upstream services (session-api, catalog-api) land before the flows that depend on them, visualized as Jira blockers. Third, risk and complexity — high-risk work like the saga compensation logic gets scheduled early so we have time to iterate; I learned that the hard way when Disney+'s async provisioning broke our synchronous flow late in the project. Day to day I keep a living backlog with acceptance criteria on every story and run a weekly triage with the product owner. The core principle: requirements aren't static — I re-prioritize every sprint based on what we've learned.

## 3. What experience do you have with BI tools like Tableau or Power BI?
My primary focus was systems analysis, but data analysis and visual storytelling were central to it. I queried PostgreSQL — subscriptions, orders, audit logs — to analyze lifecycle patterns, like which merchants had the highest provisioning failure rates, which directly informed our per-provider circuit-breaker thresholds. I built the same kind of decision artifacts a BI dashboard produces: latency-budget tables of P99 times across every service hop, error-distribution analysis across four defined error tiers, and cost/complexity matrices comparing integration approaches for stakeholders. The underlying skills — SQL querying, data modeling, metric definition, visual storytelling — are exactly what BI tools formalize, so I pick up any specific tool quickly; the domain changes, the analytical approach transfers.

## 4. How do you handle conflicting priorities across multiple projects?
On Bell I constantly juggled active integrations, platform refactors, and production defect triage. Four things: (1) I make conflicts visible with a simple priority matrix — workstream, deadline, status, stakeholder — shared so everyone sees the same picture. (2) I negotiate with data, not opinions — when two teams needed me the same sprint, I showed the concrete dependency schedule ("Disney+ spec by Tuesday, saga design by Thursday, I can do both in this order") and the conflict usually resolved itself. (3) I protect focused time — deep analysis in the mornings, meetings and triage in the afternoons; that's how I delivered the Radio-Canada spec while supporting Disney+ UAT. (4) I escalate early — if I genuinely can't hit everything, I bring my manager a clean trade-off: "A and B, or A and C — which?" I never silently miss a deadline.

## 5. What's your approach to data analysis as a BSA?
Three layers. Field-level tracing: for every integration I map each field UI → GraphQL variable → Go struct field → DB column, which exposes transformations and missing fields before development. Event/payload analysis: I verify Kafka payloads carry everything downstream consumers need — I once caught a missing billingCycle field in the OrderCreated event during analysis rather than in production. Operational analysis: I mine production logs, error rates, and latency to drive requirements — analyzing P99 provisioning latency showed Bango was consistently slower (1.2s vs Netflix's 400ms), which led to provider-specific timeout config, and DLQ analysis surfaced a Disney+ callback schema mismatch. The throughline: I don't just document what the system should do — I verify what it actually does with real data, and use that to improve the design.

## 6. How do you ensure successful UAT?
I make UAT a natural extension of requirements, not a separate phase — every scenario traces back to an acceptance criterion I wrote. Four stages: (1) AC-driven test plans — each Given-When-Then criterion becomes a test case, so QA never has to guess what to test. (2) Test data prep — valid CPM accounts for the happy path, expired DynamoDB sessions, merchant stubs configured to fail for circuit-breaker testing, specific PostgreSQL states for grace-period and reversal cases. (3) Environment validation — I confirm all 5 adapters are deployed, Kafka topics configured, TTLs and breaker thresholds match production; I learned this after a UAT cycle failed on mismatched breaker settings. (4) Defect triage with traceability — every defect maps back to the AC, orchestration step, and data-mapping field, which cut our average resolution time from 2 days to half a day. The principle: UAT validates requirements, it shouldn't discover them.

## 7. How do you communicate complex technical information to non-technical stakeholders?
Analogies, visuals, and progressive disclosure. When I explained circuit breakers to the VP of Product, I used a household fuse: "When Netflix has a problem, our fuse trips — for 30 seconds Netflix orders fail fast but every other provider keeps working, then we retest." He got it instantly and even proposed a UX improvement. I match the visual to the audience — system context diagrams for executives, state machines for product owners, sequence diagrams for developers: same system, three levels of detail. And I disclose progressively — start with "writes and reads take different paths, so there's a sub-5-second delay before you see a change," and only unpack the CQRS pipeline if they ask why. The meta-principle: I translate technical detail into business impact. Nobody needs to know we use a saga — they need to know "if the merchant fails, we automatically undo the order and the customer isn't charged."

## 8. How do you stay updated with industry trends?
Mostly by applying, not just reading. The Bell platform itself was a continuous learning environment — 60+ Go microservices exposed me to event-driven architecture, saga, CQRS, circuit breakers, and cloud-native patterns, and each new merchant brought new problems (Disney+ pushed me into webhook and polling-fallback patterns). I study integration patterns through Martin Fowler, Microservices.io, and the AWS Well-Architected Framework — when we debated orchestration vs choreography I researched both and recommended orchestration because we needed explicit compensation. I also build my own software projects to stay hands-on with modern frameworks and tooling. And I do cross-domain mapping — subscription lifecycle (PENDING → ACTIVE → CANCELLED) maps cleanly onto benefits enrollment (APPLIED → ENROLLED → TERMINATED), which helps me recognize patterns faster in a new domain.

## 9. Describe your experience with Agile methodologies.
I worked in Scrum throughout Bell as the primary BSA embedded in a cross-functional team. In planning I present prioritized stories with complete ACs, data mappings, and orchestration specs, and lead pointing by relative complexity ("Radio-Canada is like Netflix but HMAC auth, so add 3 for adapter complexity"). In refinement I use "progressive decomposition" — present what we know, list unknowns as time-boxed spike stories, and shift from "estimate this" to "what must we learn before we can estimate?" In standups I flag analysis blockers and coordinate UAT readiness with QA; in reviews I demo against ACs; in retros I contribute process fixes — I introduced BSA peer review of requirements docs after inconsistencies between mappings and ACs caused rework. My Definition of Done includes AC verified, data mapping validated, error scenarios tested, and Kafka events confirmed. For a BSA, "working software over documentation" means requirements complete enough that devs don't guess, concise enough that they actually read them.

## 10. How do you gather and document business requirements?
It's iterative and evidence-based. Step 1 — stakeholder discovery: for a new merchant that meant the product owner, the merchant's tech team, the backend lead, QA, and ops, each contributing a different dimension. Step 2 — current-state analysis: I trace the actual code paths before writing anything new, which surfaces implicit requirements stakeholders forget, like per-merchant breaker config or DLQ retry policy. Step 3 — documentation: four linked artifacts — a BRD for business-level needs, field-level data mappings, orchestration specs with error handling and compensation per step, and Given-When-Then acceptance criteria covering happy path, errors, edge cases, and async behavior. Step 4 — validation: I review with devs (implementable?), QA (testable?), and the product owner (right intent?), with a BSA peer review before grooming. Step 5 — living documentation: when a dev finds an edge case mid-build, I update the AC and mapping. The docs are never "done" — they're current.

## 11. What techniques do you use to identify process improvement?
Three. Data-driven friction analysis: when defect resolution averaged 2 days, I traced it to devs being unable to connect defects to requirements — I introduced AC-referenced defect tickets ("violates AC-1.3: status should be PENDING, actual NULL") and cut it to half a day. Pattern recognition across integrations: after three merchants I saw everyone documenting the same pattern differently, so I built standard templates (data mapping, orchestration spec, AC checklist) that cut documentation time ~40% on the next two onboardings. Retrospective feedback loops: devs wanted error scenarios up front, so I made an "Error Scenarios" section mandatory across all four error tiers, which reduced mid-sprint clarification requests ~60%. The one I'm proudest of is the BSA peer-review process — a completeness checklist before docs reach dev — which caught about 30% of issues before development.

## 12. How do you keep a project on schedule and within budget?
Four levers. Front-loading analysis to prevent rework — complete mappings, specs, and error scenarios mean devs implement without guessing; stories with complete specs hit their estimates, while early stories without them overran 30–50% on clarification cycles. Dependency mapping and critical path — I sequence stories by dependency, and when Disney+ went async mid-project I immediately mapped the impact (3 stories to revise, 2 new) and gave the product owner two options with timelines rather than a late surprise. Risk-based scheduling — I start high-risk external integrations early; I pushed the Disney+ spike 2 sprints ahead of target, which is exactly why we caught the async change in time. Transparent status — a weekly view of each workstream vs plan, and when something slips I bring "here's the impact and here's what we can do," never just "we're behind." A BSA can't control the budget directly, but requirement quality drives accurate estimates and prevents costly rework.

## 13. Give an example of using data analysis to improve a business system.
After our first month live with Netflix and Disney+, ops noticed ~2% of orders were stuck in PENDING forever — never ACTIVE, never FAILED. I queried subscriptions (PENDING, created_at > 7 days), joined to orders, and checked the audit log for the last recorded action. These orders had written to PostgreSQL (saga step 2) but the merchant provisioning call (step 3) had timed out — and the code was silently swallowing the timeout instead of triggering compensation. The Kafka publish (step 4) never fired, so nothing hit the DLQ either. Correlating by time window, the failures clustered around Netflix's Sunday 2–4am maintenance. That analysis drove three requirements: provisioning timeouts must trigger saga compensation, a fallout-process Lambda must scan for stuck orders and auto-remediate, and the breaker must use provider-specific timeouts (Netflix 800ms, Disney+ 1.2s, Bango 1.5s). Stuck orders dropped from 2% to 0.01%.

## 14. How do you handle changes in scope or client requirements?
Structured impact assessment, transparent communication, documented trade-offs. Two weeks before launch, Disney+ told us their provisioning API would be async, not sync — and our whole flow assumed a synchronous activationUrl in the same response. Within 24 hours I mapped the change against every artifact: a new "await callback" orchestration step, a new PROVISIONING state, three ACs to revise, a changed Kafka sequence, and a new UI "processing" screen. Then I gave the product owner two options: (A) delay launch 3 weeks for full webhooks, or (B) launch with a polling fallback while we build webhooks in parallel. I recommended B with explicit cutover criteria — ship polling now, switch to webhooks once the handler passes UAT, and keep polling as the fallback for any future provider without webhooks. I revised all affected docs within 2 days. We launched on schedule; webhooks shipped two sprints later. The key was responding with a concrete impact assessment and actionable options, not just flagging risk.

## 15. What's your experience creating and maintaining technical documentation?
It's a core deliverable. On Bell I maintained six living document types: field-level data mappings (UI → GraphQL → Go → DB → Notes), orchestration specs (service, action, payload, error handling, compensation per step), annotated OpenAPI specs for 9 services plus 2 gRPC protos, JDL-style domain models for the core entities, state-machine diagrams of the subscription lifecycle, and an error catalog of 17 scenarios across 4 tiers with status codes, UI behavior, and recovery paths. I treat documentation like code — never "done," always "current" — reviewing and updating affected docs every sprint, with a BSA peer review against a completeness checklist before anything reaches the dev team.

## 16. How do you approach risk management?
Every risk should map to a corresponding requirement or architectural decision — otherwise you've only acknowledged it, not mitigated it. I categorized Bell's risks into four tiers. External dependency (highest): each merchant could fail independently, so I wrote requirements for per-merchant circuit breakers (open after 5 failures in 30s) plus a DLQ with auto-remediation. Data consistency: CQRS means read/write can diverge, so I documented the sub-5-second eventual-consistency window and required idempotent operations. Integration change: partners can change contracts — which Disney+ did — so I pushed the adapter pattern and an anti-corruption layer around legacy CPM early, isolating changes to one provider. Knowledge/process: single points of knowledge, mitigated with standardized templates and peer review, so a junior BSA could independently run the next integration from my Radio-Canada templates.

## 17. Describe your process for developing BSA reports.
A structure that moves from business context to technical detail so it serves both audiences: (1) Business problem statement — the need in plain terms. (2) Current-state analysis — how it actually works today, traced from the codebase rather than trusting stale docs. (3) Gap analysis — for Radio-Canada that was no HMAC adapter, no handling for their 5-day grace window, no mapping for their catalog format. (4) Proposed solution as concrete artifacts — data mapping, orchestration spec, Given-When-Then ACs, state-machine updates. (5) Impact assessment — services affected, testing scope, dependencies, risks. (6) Recommendations — always at least two prioritized options with trade-offs so stakeholders decide from information, not a mandate. Each report lives in the team wiki and is updated every sprint as implementation reveals detail.

## 18. What tools do you consider essential for a BSA?
Five categories. Requirements management — Jira for stories, ACs, and traceability links (BR → Story → Test), Confluence for living docs. Diagramming — Mermaid embedded in markdown so sequence diagrams, state machines, and ERDs are version-controlled alongside the code; Lucidchart/draw.io for heavier visuals. API and data analysis — Postman to validate actual responses against the OpenAPI spec, and SQL clients against PostgreSQL (querying the audit log is how I found the stuck-orders issue). Codebase navigation — VS Code and Git; I don't write production code but I read it constantly to trace data flows and confirm implementation matches requirements. Communication — Slack, screen sharing, and whiteboarding (the fuse analogy was a whiteboard moment). But the most important "tool" isn't software — it's the standardized artifact templates that let any BSA produce consistent, high-quality deliverables.

## 19. How do you ensure scalability of the systems you work on?
By making non-functional requirements explicit and testable, not assumed. Bell went from 2 merchants to 5 in a year with more planned. Adapter pattern for merchant scalability — I specified "adding a merchant must not require changes to reseller-service core logic," so each integration was just 4 interface methods and a new service; Radio-Canada took 2 weeks as proof. CQRS for read/write scalability — separate paths so read services scale independently during promotion spikes. Event-driven for throughput — Kafka with a non-blocking publish requirement so consumers scale without touching the core transaction path. Latency budgets — documented P99 targets per hop as the baseline for capacity planning. And per-merchant circuit breakers so one provider's failure never cascades. Scalability requirements should be explicit and testable, not hand-waved.

## 20. Explain the difference between a BRD and an SRS.
They serve different audiences at different altitudes, and on Bell I produced both, tightly linked. The BRD is the what and why — for business stakeholders, in business language: "Bell residential customers must be able to add, manage, and cancel streaming subscriptions with charges on their existing account, and new providers must be onboardable within 4 weeks." The SRS is the how — for developers, architects, and QA, in technical specifics: "submitSubscription must validate sessionId via session-api, write a PENDING record to PostgreSQL, call merchant-api-<providerId>.Provision synchronously, publish OrderCreated to Kafka fire-and-forget, and log ORDER_PLACED to audit-api with correlationId; P99 500ms–2s." They connect through traceability — one BRD requirement decomposes into the data mapping, orchestration spec, ACs, and error handling in the SRS, linked in Jira as BR → Epic → Story → Test. The BRD is the stable anchor; the SRS evolves as the design matures.

## 21. How do you facilitate communication between business and technical teams?
I'm the translator, and my main tools are shared artifacts, structured workshops, and real-time translation. The data mapping is common ground — when the product owner said "the customer's plan should show on the confirmation screen," I could point to the exact field (selectedProductId → subscriptions.product_id, populated at qualification): the dev knew where to look, the PO confirmed it was right, same artifact two perspectives. For cross-team disputes I run workshops on a shared picture — when billing and engineering disagreed on status-visibility latency, I asked billing "how soon do you need changes?" (30s) and engineering "what's the current lag?" (under 5s), and the conflict dissolved on the numbers. And I translate live in meetings — a dev's "race condition between the consumer and the direct API call" becomes "there's a brief window where the customer might see stale data; we'll add a cache refresh."

## 22. What strategies do you use to prioritize project requirements?
A multi-dimensional framework. MoSCoW with stakeholder sign-off — for launch, Must-Haves were add/cancel/view subscription; grace-period and reversal were Should-Haves; promo pricing a Could-Have. Dependency-driven ordering — session-api and catalog-api had to precede the order-submission flow, sequenced in Jira. Risk-weighted scheduling — high-risk external integrations and new patterns go early; I moved the Disney+ spike up 2 sprints, which is why we caught the async requirement in time. Value-vs-effort quadrants for items of similar priority — quick wins first, then strategic investments, defer low-value/high-effort. And contractual/regulatory constraints — fixed merchant launch dates are non-negotiable top priority. Prioritization is continuous — I re-rank every sprint and keep the rationale visible so decisions stay transparent.

## 23. Describe a time you had to mediate stakeholder conflicts.
On the Radio-Canada integration, the product team wanted a fast custom build to hit a 6-week contractual deadline, while engineering — fresh off cleaning up debt from a previous rushed integration — insisted on the full standard adapter interface at ~4 weeks, leaving almost no test buffer. Both concerns were legitimate. I reframed it from "fast vs right" to "what's the minimum scope that satisfies both?" I put up a side-by-side: Option A (custom, 2 weeks, but new tech debt) vs Option B (full adapter, 4 weeks). Then I proposed Option C — a phased adapter: implement Provision and Deprovision first (2.5 weeks, covering 90% of launch needs — customers can add and cancel), and defer CheckStatus and HandleCallback to a fast-follow (1.5 weeks post-launch). The adapter pattern is used from day one, but scope is trimmed. In a joint meeting both sides agreed: product got its date with a QA buffer, engineering got architectural consistency with no debt. Radio-Canada launched on time; the deferred methods shipped two sprints later. The key was addressing the underlying constraints, not the surface positions.

## 24. How do you approach testing and quality assurance?
I build quality into requirements rather than bolting it on. Requirements-driven planning — every Given-When-Then AC becomes a test case; "Add Subscription" had 6 ACs that generated 15+ scenarios across happy path, the four error tiers, edge cases, and async verification. Data-mapping validation — after deploy I trace real API responses field by field: correct subscriptionId, status=PENDING, correct billingAccountNumber in the OrderCreated event; this catches field-level bugs unit tests miss. Error-scenario verification — fixtures simulate failure at each step (merchant 500 opens the breaker, expired session returns 404, failed Kafka publish still commits the order and routes to DLQ), 17 scenarios as QA's negative test plan. Contract testing — I verify actual behavior against the OpenAPI spec; we caught 3 contract mismatches in review that would've failed in production. Plus UAT support — scripts from ACs, test data, environment validation, and AC-level defect triage.
